If you just want to play with friends while you're active yourself - use the co-op menu in the client.
If you want to leave a server on while your pc is active, use the colonyserver executable of your install manually and follow the menus.
Both of these options require you to either leave the client or server active, and require steam to be started and logged in

If you're looking to host a headless server on a VPS or NAS or such, the easiest is probably to use the included docker(-compose) files in gamedata/help/hosting_docker/
There's tutorials out there for how-to with docker
The easiest path is to copy docker-compose.yml and server.config.json from gamedata/help/hosting_docker/ into their own folder and then run docker compose up -d there.
The compose file uses the published image pipliz/colony-survival-dedicated-server:latest from Docker Hub, so you do not need to build the image locally unless you want to customize it.
Rough outline: it uses SteamCMD to install appid 748090 (the dedicated server) into the persistent /server volume on first start.
Both steamCMD and the game install are persisted outside of the image.
entrypoint.sh deals with this.
The container is just a wrapper that starts the server using the server.config.json file.
The included compose file mounts ./server.config.json to /server/server.config.json read-only, which is the intended way to customize server settings.
If the mounted config file exists, it is used as-is. If it does not exist, the container falls back to its built-in default config.

To do it manually, you can lookup general steamCMD guides, use appid 748090.

Some notes on the config file (the preferable way to pass configs to the server)

Default config file only has NewOptions - if the used WorldName already exists it transforms into a loadworld. If NewOptions is not used, you can use LoadOptions with the same WorldName key.
"DifficultyKey" indexes the difficulty presets in gamedata/settings/server.json
Seed can be either text or integer (text is hashed to an integer by the server)
"ServerIP" can generally be left at 0.0.0.0, but can be relevant if there are multiple network interfaces - the IP configured should be the one attached to the network interface desired.
